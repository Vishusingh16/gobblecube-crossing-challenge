1. The Problem: What Are We Building?

1.1 The Real-World Scenario

Imagine a self-driving delivery robot (like an Amazon Scout) driving down a sidewalk at walking speed. A pedestrian is standing near the curb. The robot needs to answer two questions in real-time:

Question 1 (Intent): Will this pedestrian cross the road in the next 2 seconds? (Yes/No, as a probability)

Question 2 (Trajectory): If they DO move, exactly WHERE will they be at 0.5s, 1.0s, 1.5s, and 2.0s from now? (as bounding box coordinates)

Think of it like this: You're driving a car and see someone near a crosswalk. Your brain instantly judges "are they going to step out?" and "which way are they heading?" We're teaching a computer to do the same thing.

1.2 Why Does This Matter?

This is a life-safety problem for autonomous vehicles. If the robot mispredicts:

  • False negative (says they won't cross, but they do) = collision risk
  • False positive (says they will cross, but they don't) = unnecessary braking, bad UX
  • Bad trajectory (wrong position prediction) = wrong evasion maneuver

1.3 What Gobblecube Gave Us

Gobblecube (an AI company that builds autonomous delivery robots) created this as a hiring challenge. They gave us:

  • Training data: 28,680 examples of pedestrian tracklets (short video clips turned into number sequences)
  • Dev data: 6,065 examples for testing our model
  • A scoring formula: That combines both questions into one number (lower = better)
  • A baseline: A working but basic solution scoring 0.83 on Dev

Our job: Beat the baseline by as much as possible.

2. Key Concepts Explained (For Non-ML People)

2.1 What is Machine Learning? (The Restaurant Analogy)

Imagine you're a restaurant critic. You've eaten at 1,000 restaurants and noticed patterns:

  • Dirty menu = bad kitchen = probably bad food
  • Long line outside = probably good food
  • Fancy neighborhood + high prices = probably fancy food

Now someone asks you to rate a NEW restaurant you've never been to. You use your learned patterns to make a prediction. That's machine learning.

Machine Learning = Teaching a computer to find patterns in data, so it can make predictions on NEW data it hasn't seen before.

The "learning" part = The computer adjusts its internal settings (called "weights" or "parameters") by looking at thousands of examples, until its predictions match the correct answers as closely as possible.

2.2 Key Terms You'll See Everywhere

Model

A mathematical formula that takes inputs and produces outputs. Think of it as a black box:

Inputs (pedestrian's past 16 positions) --> [MODEL] --> Outputs (will they cross? where will they be?)

Like a function in programming: model(pedestrian_data) = prediction

Features

The inputs to the model. Raw data is transformed into numerical features the model can understand.

Analogy: When you judge a restaurant, your "features" are: cleanliness (1-10), wait time (minutes), price ($), location rating (1-5). These are numbers that describe the thing you're predicting.

In our project: Our features include things like: pedestrian's X position (0-1920 pixels), Y position (0-1080 pixels), speed (pixels per frame), direction (radians), etc. We created 53+ numerical features from the raw bbox history.

Training (Fitting)

The process of showing the model thousands of examples so it can learn patterns. Like studying for an exam by doing practice problems.

Training loop:
1. Show model an example (input features + correct answer)
2. Model makes a prediction
3. Compare prediction to correct answer (calculate "loss")
4. Adjust model's internal settings to reduce the error
5. Repeat 28,680 times (once per training example)
6. Repeat the whole process multiple times ("epochs")

Prediction / Inference

Using the trained model to make predictions on new data. Like taking the actual exam after studying.

Loss Function

A number that measures how wrong the model's predictions are. Lower = better. The model tries to minimize this during training.

  • BCE (Binary Cross-Entropy): Measures how bad the intent probability is. Used for yes/no questions.
  • MSE (Mean Squared Error): Measures how far off the predicted position is from the actual position.
  • ADE (Average Displacement Error): The average pixel distance between predicted and actual bbox centers.

Overfitting vs Underfitting

Overfitting = Memorizing the training data instead of learning patterns. Like a student who memorizes answers but can't solve new problems. The model scores great on training data but badly on new data.

Underfitting = Model is too simple to learn the patterns. Like a student who didn't study enough.

Our approach: We use "regularization" (penalizing complexity), "early stopping" (stop training before overfitting), and a dev set to check for overfitting.

Gradient Boosted Trees (GBT)

The type of ML model we used. Think of it as a committee of decision trees that vote together:

  • Decision tree: A flowchart of yes/no questions. "Is speed > 5? If yes, is Y > 500? If yes, predict 'crossing'."
  • Boosted: Each tree focuses on the mistakes of the previous trees. Like having a study group where each person specializes in different weak areas.
  • XGBoost / LightGBM: Two popular implementations. XGBoost is more careful, LightGBM is faster.

Ensemble

Combining multiple models' predictions for better accuracy. Like asking 3 doctors instead of 1.

Model A says: "70% chance of crossing"
Model B says: "80% chance of crossing"
Ensemble:      "75% chance of crossing" (average)

Residual Learning

Instead of predicting the FINAL answer, predict the CORRECTION to a simple baseline:

Baseline (constant velocity): "Pedestrian will be at position (500, 300) in 2 seconds"
Our model correction:           "Actually, they're decelerating, so subtract 15 pixels"
Final prediction:               "Position (485, 300)"
Why this works: The baseline is already pretty good for short horizons. The model only needs to learn the small corrections, which is MUCH easier than learning the full prediction from scratch.

Bounding Box (bbox)

A rectangle drawn around an object in an image. Defined by 4 numbers: [x1, y1, x2, y2]

   x1,y1 -------- x2,y1
     |              |
     |  PEDESTRIAN  |
     |              |
   x1,y2 -------- x2,y2

x1 = left edge, y1 = top edge, x2 = right edge, y2 = bottom edge
Center = ((x1+x2)/2, (y1+y2)/2)
Width = x2 - x1, Height = y2 - y1

4. How We Scored: The Evaluation Metric

4.1 The Formula

composite = 0.5 * (intent_BCE / 0.2488) + 0.5 * (mean_pixel_ADE / 49.80) Where: - intent_BCE = Binary Cross-Entropy of our crossing probability - mean_pixel_ADE = Average pixel distance between predicted and actual bbox centers - 0.2488 and 49.80 are normalization constants (so "do nothing" = 1.0) - Lower is better, 0.0 = perfect prediction

4.2 What Each Term Means

Intent Term (BCE)

Binary Cross-Entropy measures how "surprised" the scoring formula is by our prediction. If we say "90% chance of crossing" and they DO cross, that's good (low surprise). If we say "10% chance" and they DO cross, that's bad (high surprise).

BCE = -[y * log(p) + (1-y) * log(1-p)]

Where: y = actual answer (0 or 1), p = our predicted probability

Perfect prediction: BCE = 0.0
Always predict 50/50: BCE = 0.693
Always predict class average (8%): BCE = 0.249

Trajectory Term (ADE)

Average Displacement Error = average pixel distance between our predicted bbox center and the actual bbox center, across all 4 horizons.

For each horizon (0.5s, 1.0s, 1.5s, 2.0s):
  ADE_h = sqrt( (predicted_cx - actual_cx)^2 + (predicted_cy - actual_cy)^2 )

mean_ADE = average of ADE_500ms, ADE_1000ms, ADE_1500ms, ADE_2000ms

5. Our Approach: Architecture Overview

5.1 High-Level Pipeline

We split the problem into TWO separate models that work independently:

INPUT | +-----------------+-----------------+ | | +----+----+ +------+------+ | INTENT | | TRAJECTORY | | MODEL | | MODEL | | (Will | | (Where will | | they | | they be?) | | cross?) | | | +----+----+ +------+------+ | | +----+----+ +------+------+ | LightGBM | | 8 XGBoost | | + | | regressors | | XGBoost | | (cx,cy per | | ensemble | | horizon) | +----+----+ +------+------+ | | v v intent: 0.73 bbox_500ms: [x1,y1,x2,y2] bbox_1000ms: [x1,y1,x2,y2] bbox_1500ms: [x1,y1,x2,y2] bbox_2000ms: [x1,y1,x2,y2]
Why two separate models? The two tasks need different approaches:
- Intent is a classification problem (yes/no) - best solved with tree-based classifiers
- Trajectory is a regression problem (predict numbers) - best solved with per-horizon regressors
Separating them lets us optimize each independently.

6. Step-by-Step: How We Solved It

Step 1: Understand the Baseline

We first ran the provided baseline to understand what "0.83" looks like:

python baseline.py    # Trains XGBoost intent classifier with 20 features
python grade.py       # Scores 0.8311 on dev

The baseline uses simple features (position, velocity, aspect ratio, ego speed) and constant-velocity extrapolation for trajectory (assume pedestrian keeps moving at current speed). This is "naive" - it doesn't account for acceleration, deceleration, or direction changes.

Step 2: Exploratory Data Analysis (EDA)

We explored the data to find patterns:

  • 8% positive rate - very imbalanced, but we must NOT rebalance (log-loss penalizes bad calibration)
  • 93% have ego data - must handle 7% with no vehicle data
  • JAAD vs PIE split - different datasets have different metadata richness
  • Zero-velocity ADE explodes at longer horizons (42px at 0.5s, 187px at 2s) - trajectory is the harder problem

Step 3: Build Feature Engineering

Created 53 features from raw data, organized into groups:

Step 4: Train Intent Model

We tried multiple approaches and learned from failures:

  1. XGBoost with class weighting - BCE 0.30 (worse than baseline! Over-predicts positives)
  2. LightGBM without class weighting - BCE 0.24 (much better! Proper calibration)
  3. LightGBM + XGBoost ensemble - BCE 0.23 (best! Averaging smooths out errors)
Key lesson: For log-loss, you need CALIBRATED probabilities, not balanced predictions. Setting scale_pos_weight=11.6 (to match the 11.6:1 class ratio) made the model over-predict positives, which log-loss punishes heavily. Removing it and adding regularization was the fix.

Step 5: Train Trajectory Model

We tried three approaches:

  1. LSTM neural network - ADE 39.2px (barely better than baseline 39.6px. Dataset too small for deep learning.)
  2. XGBoost per-horizon regressors - ADE 29.1px (27% improvement! Tabular features + GBT wins on small data.)
  3. XGBoost + LightGBM ensemble - ADE 28.8px (marginal improvement, doubles inference time. Not worth it.)
Key lesson: With only 29K samples, classical ML (gradient boosted trees) beats deep learning (LSTM). Deep learning needs millions of examples to learn patterns that GBTs can pick up from thousands. Domain knowledge (good features) + simple model > raw data + complex model.

Step 6: Build Inference Pipeline

The predict.py file ties everything together:

8. Code Walkthrough: Key Functions Explained

8.1 Feature Extraction (features.py)

This is the brain of our system. It converts raw data into numbers the model understands.

def extract_features(req: dict) -> np.ndarray:
    # Step 1: Get the 16-frame bounding box history
    hist = stack_bbox_history(req["bbox_history"])  # shape: (16, 4)

    # Step 2: Calculate derived values
    cx = center_x(hist)            # X center of bbox each frame
    cy = center_y(hist)            # Y center of bbox each frame
    vx = velocity_x(cx)            # How fast X is changing (diff)
    vy = velocity_y(cy)            # How fast Y is changing (diff)
    speed = sqrt(vx^2 + vy^2)      # Overall speed each frame
    accel = acceleration(speed)     # How speed is changing
    angle = arctan2(vy, vx)        # Direction of movement

    # Step 3: Build feature vector (53 numbers)
    features = [
        position_normalized,        # Where are they?
        velocity_normalized,        # How fast?
        acceleration,               # Speeding up or slowing down?
        direction,                  # Which way?
        ego_interaction,            # Vehicle effect
        scene_metadata,             # Weather, time, location
        derived_features,           # Domain knowledge
    ]

    return features  # 53 numbers describing this pedestrian
Why features matter: The #1 thing that determines ML model quality is the features. A simple model with great features beats a complex model with raw data. This is where full-stack skills shine - you understand data transformation and can build expressive features.

8.4 Constant-Velocity Baseline

def constant_velocity_baseline(request):
    # Look at the last 5 frames to estimate velocity
    vx = mean(diff(cx[-5:]))    # Average horizontal velocity
    vy = mean(diff(cy[-5:]))    # Average vertical velocity

    # For each future time point, extrapolate
    for horizon_frames in [8, 15, 23, 30]:  # at 15 fps
        future_cx = current_cx + vx * horizon_frames
        future_cy = current_cy + vy * horizon_frames
        bbox = [future_cx - w/2, future_cy - h/2,
                future_cx + w/2, future_cy + h/2]

    return bboxes

11. What Failed & Why (Lessons Learned)

Failure 1: LSTM Trajectory Model

What we tried: Bidirectional LSTM with attention pooling (389K parameters)
Result: ADE 39.2px (baseline was 39.6px - barely any improvement)
Why it failed: Only 29K training samples. Neural networks need millions of examples to learn patterns. With small data, the LSTM memorized noise instead of learning real patterns.
Lesson: Classical ML (GBT) beats deep learning on small tabular datasets. Always start simple.

Failure 2: Class Weighting for Intent

What we tried: scale_pos_weight=11.6 (inverse of class ratio)
Result: BCE jumped to 0.30 (worse than baseline!)
Why it failed: Class weighting makes the model predict more positives, but log-loss PENALIZES overconfident wrong predictions. The model started saying "30% chance of crossing" for non-crossing pedestrians.
Lesson: For probabilistic metrics (log-loss, BCE), you need calibrated probabilities, not balanced predictions.

Failure 3: Trajectory Ensemble

What we tried: Averaging XGBoost + LightGBM trajectory predictions
Result: ADE improved by only 0.3px, but inference time doubled
Why it failed: Both models learned similar corrections. Averaging two similar models doesn't help much (low diversity).
Lesson: Ensembles help when models make DIFFERENT errors. If they agree, the average doesn't improve anything.

"We split the problem into two independent models: an ensemble intent classifier (LightGBM + XGBoost) and per-horizon XGBoost trajectory regressors using residual learning from a constant-velocity baseline. The intent model uses 53 engineered features including position, velocity, acceleration, ego interaction, and scene metadata. The trajectory model uses 71 features to predict corrections to the baseline, achieving 29.1px ADE vs 40.2px baseline."

Q2: "What was your biggest improvement and why?"

"Switching from constant-velocity to XGBoost regressors with residual learning. The baseline assumes pedestrians maintain their current velocity, but real pedestrians accelerate, decelerate, and change direction. Our XGBoost models learn these corrections from 71 features, reducing ADE by 27%. Residual learning was key - predicting small corrections to a reasonable baseline is much easier than predicting absolute positions from scratch."

Q3: "What didn't work?"

"Three things: (1) LSTM trajectory model - the 29K dataset was too small for deep learning; GBTs won. (2) Class weighting for intent - it destroyed probability calibration and worsened BCE. (3) Trajectory ensemble - marginal ADE gain didn't justify 2x inference latency. These failures taught us to start simple and verify each assumption."

Q4: "How would you improve this with more time?"

"Four things: (1) Platt scaling or isotonic regression for intent calibration, (2) multi-output regressor to predict all 8 trajectory values jointly, (3) position-aware features encoding distance to road edge, and (4) data augmentation with random bbox transformations to make the dataset 10x larger for deep learning."

12.2 Core ML Concepts to Know

Bias-Variance Tradeoff

Bias = model is too simple (underfitting). Variance = model memorizes noise (overfitting). Our GBT models with regularization hit the sweet spot.

Feature Engineering

The art of transforming raw data into informative features. This is where domain knowledge and creativity matter most. Our 53 features encode physics (velocity, acceleration), geometry (position, aspect ratio), and context (weather, ego motion).

Cross-Validation

Splitting data multiple ways to get a reliable estimate of model quality. We used a single dev set but could have used 5-fold CV for more robust evaluation.

Hyperparameter Tuning

Adjusting model settings (number of trees, depth, learning rate) to find the best configuration. We tuned manually - in production, you'd use automated tools like Optuna or Ray Tune.

Calibration

Ensuring predicted probabilities match reality. If the model says "70% chance of crossing", then 70% of those cases should actually be crossing. Our ensemble improved calibration by averaging out individual model biases.
